## Background

Gateway will own authenticated WebSocket connections, heartbeats, online delivery, and delivery ACKs. Message Service and PostgreSQL already own durable messages, conversation seqs, and read state. Gateway needs a Redis-backed presence layer before the real WebSocket server is introduced so connection state can be shared across gateway instances.

## Goals

- Provide local Redis through `docker-compose.yml`.
- Define a presence store contract for Gateway connection lifecycle code.
- Keep default tests and service starts independent from Redis by using an in-memory implementation.
- Use Redis only for online and short-lived connection state.
- Keep PostgreSQL as the authoritative store for durable user, message, conversation, and read-state data.

## Non-goals

- Do not implement a real WebSocket server.
- Do not deliver messages or implement delivery retry queues.
- Do not store message history, `has_read_seq`, user profile, auth secrets, friendship, or group membership in Redis presence keys.
- Do not require Redis for `go test ./...`.

## Data Authority Boundary

PostgreSQL remains the source of truth for durable data:

- user profile and logical user existence;
- auth credential rows, owned by Auth Service;
- friendships and group memberships;
- message history, `server_msg_id`, conversation `seq`, conversation `max_seq`;
- `user_id + conversation_id -> has_read_seq`.

Redis presence is non-authoritative runtime state. If Redis is lost, clients reconnect and Gateway rebuilds presence from active WebSocket sessions. Redis presence loss must not change message history, unread counts, read receipts, membership, or account data.

## Contract

Code lives under `internal/presence/`.

```text
RegisterConnection(ctx, metadata, ttl) error
Heartbeat(ctx, user_id, connection_id, ttl) error
UnregisterConnection(ctx, user_id, connection_id) error
ListUserConnections(ctx, user_id) ([]ConnectionMetadata, error)
IsUserOnline(ctx, user_id) (bool, error)
```

`ConnectionMetadata` records the runtime connection identity and routing hints:

```text
user_id
connection_id
gateway_id
device_id
platform
remote_addr
user_agent
connected_at
last_heartbeat_at
expires_at
```

`connection_id` is generated by the future Gateway process and is scoped globally enough to route or unregister one WebSocket connection. `gateway_id` identifies the Gateway instance that owns the socket.

## Redis Keys

Default prefix:

```text
agents_im:presence
```

Keys:

```text
agents_im:presence:user:{user_id}:connections -> Redis set of connection_id values
agents_im:presence:user:{user_id}:online      -> short-lived marker, value "1"
agents_im:presence:conn:{connection_id}       -> Redis hash of ConnectionMetadata
```

Hash fields:

```text
user_id
connection_id
gateway_id
device_id
platform
remote_addr
user_agent
connected_at_unix_ms
last_heartbeat_at_unix_ms
expires_at_unix_ms
```

`IsUserOnline` and `ListUserConnections` verify live connection hashes instead of trusting only the online marker. Stale set members are removed when listed.

## TTL and Heartbeat

Default heartbeat TTL is 60 seconds. The caller passes TTL per register and heartbeat so Gateway can tune heartbeat cadence later.

Recommended phase-1 behavior:

- `RegisterConnection` writes the connection hash with heartbeat TTL, adds it to the user connection set, and refreshes the user online marker.
- `Heartbeat` refreshes the connection hash TTL, `last_heartbeat_at`, `expires_at`, the user connection set TTL, and the user online marker TTL.
- `UnregisterConnection` deletes the connection hash, removes the connection from the user set, and deletes online state if no live connections remain.
- Connection hashes expire after one missed heartbeat window.
- User set and online marker use `2 * heartbeat_ttl` as cleanup grace.

The authoritative online answer is "the user has at least one non-expired connection hash." The marker exists for cheap future fanout checks, but stale markers do not override connection verification.

## Configuration

Local Redis is provided by docker compose:

```bash
docker compose up -d redis
```

Development environment variables:

```text
REDIS_ADDR=localhost:6379
REDIS_PASSWORD=agents_im_redis_dev_password
REDIS_DB=0
PRESENCE_DRIVER=memory
PRESENCE_TTL_SECONDS=60
PRESENCE_KEY_PREFIX=agents_im:presence
```

Presence defaults to the memory store. Set `PRESENCE_DRIVER=redis` only when testing Gateway/presence behavior with local Redis.

## Implementation

- `internal/presence/store.go` defines the public contract and metadata.
- `internal/presence/memory.go` provides the default test implementation.
- `internal/presence/redis.go` uses `github.com/redis/go-redis/v9`.
- `internal/config/config.go` parses `Redis` and `Presence` config plus `REDIS_*` and `PRESENCE_*` env fallbacks.

Redis integration coverage is opt-in. `internal/presence/redis_integration_test.go` skips unless `REDIS_ADDR` is set, so ordinary `go test ./...` does not require Docker or Redis.

## Risks and Mitigations

- Stale online state after Gateway crash: heartbeat TTL expires connection hashes; list operations remove stale set members.
- Redis outage: Gateway should treat presence as unavailable and rely on reconnect and message pull compensation. Durable message and read state remain in PostgreSQL.
- Multi-connection users: online state stays true while any verified connection hash remains live.
