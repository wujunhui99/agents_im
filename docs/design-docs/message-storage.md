# Message Storage Design

Status: Implemented for PostgreSQL phase 1
Owner: IM backend
Related product spec: [`../product-specs/message-storage.md`](../product-specs/message-storage.md)
Related contract: [`./message-chain-contract.md`](./message-chain-contract.md)

## Background

Message service phase 1 owns synchronous send, pull, and read-state behavior. This document defines the future PostgreSQL and Redis storage contract behind that behavior so storage work can proceed independently from API/RPC contract work.

Current implementation status: PostgreSQL schema, migration tooling, and repository implementations are available for phase 1. Redis remains a non-authoritative future cache and is not implemented in this branch.

The storage layer must preserve these invariants:

- one accepted message is stored exactly once for a given `sender_id + client_msg_id`;
- each conversation has a gap-free, monotonically increasing `seq`;
- users can pull messages by `conversation_id + seq` range;
- `has_read_seq` only advances and never moves backward.

## Goals

- Define PostgreSQL tables for `messages`, `conversation_threads`, `user_conversation_states`, and idempotency records.
- Define the transactional outbox boundary used as the reliable event source after message acceptance.
- Define unique constraints required for idempotency, message identity, and per-conversation ordering.
- Define sequence allocation and transaction boundaries for send, pull, and mark-as-read.
- Define Redis keys that can be introduced later without changing repository behavior.
- Keep the storage contract independent from `api/message.api`, `proto/message.proto`, and HTTP handler implementation.

## Non-goals

- Implement HTTP handlers, RPC handlers, gateway delivery, Kafka transfer, push notification, or WebSocket ACK.
- Implement Kafka producer/consumer, Message Transfer worker, Gateway push fanout, or delivery ACK.
- Decide distributed sharding or cross-region replication.
- Implement message recall/delete, edits, reactions, per-device sync state, or media storage.
- Make Redis authoritative for durable history.

## Storage Ownership

PostgreSQL is the source of truth for durable messages, conversation max seq, and user read state.

Redis may cache hot data, reserve short-lived idempotency keys, and accelerate pull reads, but Redis loss must not corrupt message history or read state. Any Redis value must be rebuildable from PostgreSQL or safely treated as a cache miss.

## PostgreSQL Model

### messages

Stores immutable accepted message records.

```sql
create table messages (
  server_msg_id text primary key,
  client_msg_id text not null,
  sender_id text not null,
  conversation_id text not null,
  seq bigint not null,
  chat_type text not null,
  receiver_id text not null default '',
  group_id text not null default '',
  content_type text not null,
  content jsonb not null,
  message_origin text not null default 'human',
  agent_account_id text not null default '',
  trigger_server_msg_id text not null default '',
  agent_run_id text not null default '',
  allow_recursive_trigger boolean not null default false,
  payload_hash text not null,
  send_time timestamptz not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

Required constraints:

```sql
alter table messages
  add constraint messages_conversation_seq_uniq unique (conversation_id, seq),
  add constraint messages_sender_client_msg_uniq unique (sender_id, client_msg_id),
  add constraint messages_chat_type_check check (chat_type in ('single', 'group')),
  add constraint messages_seq_positive_check check (seq > 0),
  add constraint messages_content_type_check check (content_type <> '');

create index messages_conversation_seq_idx
  on messages (conversation_id, seq);

create index messages_sender_created_idx
  on messages (sender_id, created_at desc);
```

Field notes:

- `server_msg_id` is generated by message service or repository before insert and remains globally unique.
- `client_msg_id` is caller-provided and only unique together with `sender_id`.
- `message_origin` is one of `human`, `ai`, or `system`. AI messages must set `agent_account_id=sender_id` and may carry `trigger_server_msg_id`, `agent_run_id`, and `allow_recursive_trigger`; human/system messages do not carry Agent metadata.
- `payload_hash` is the canonical hash of idempotency-relevant payload fields. At minimum it covers `sender_id`, `client_msg_id`, `conversation_id`, `chat_type`, `receiver_id`, `group_id`, `content_type`, normalized `content`, `message_origin`, and Agent metadata.
- `content` is `jsonb` to preserve future content types. Phase 1 text messages can be encoded as `{"text":"..."}` or another contract-defined JSON shape.
- Message rows are append-only in phase 1. Future recall/delete should add tombstone fields or side tables instead of changing seq semantics.

### conversation_threads

Stores conversation-wide ordering state and last message summary.

```sql
create table conversation_threads (
  conversation_id text primary key,
  chat_type text not null,
  single_user_a text not null default '',
  single_user_b text not null default '',
  group_id text not null default '',
  max_seq bigint not null default 0,
  last_message_id text not null default '',
  last_message_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

Required constraints:

```sql
alter table conversation_threads
  add constraint conversation_threads_chat_type_check check (chat_type in ('single', 'group')),
  add constraint conversation_threads_max_seq_check check (max_seq >= 0),
  add constraint conversation_threads_single_shape_check check (
    (chat_type = 'single' and single_user_a <> '' and single_user_b <> '' and single_user_a < single_user_b and group_id = '')
    or
    (chat_type = 'group' and group_id <> '' and single_user_a = '' and single_user_b = '')
  );

create unique index conversation_threads_single_pair_uniq
  on conversation_threads (single_user_a, single_user_b)
  where chat_type = 'single';

create unique index conversation_threads_group_uniq
  on conversation_threads (group_id)
  where chat_type = 'group';
```

`conversation_id` remains deterministic:

```text
single:{lower_user_id}:{higher_user_id}
group:{group_id}
```

The storage layer should expose a single helper for this derivation so handlers, logic, and repository code do not duplicate string formatting.

Conversation ID components are validated before storage:

- `sender_id`, `receiver_id`, and `group_id` must be non-empty canonical service IDs, 128 characters or fewer.
- Components must not contain `:` because it is the `conversation_id` delimiter.
- Components and `client_msg_id` must not contain NUL because memory idempotency/read-state keys and PostgreSQL text storage must not accept hidden delimiters.
- The derived `conversation_id` must be 256 characters or fewer so a successfully sent message can later be pulled through the public conversation APIs.

Current User and Group services generate `usr_...` and `grp_...` IDs, and user-facing `identifier` values are not used as conversation ID components.

### user_conversation_states

Stores per-user read state and conversation visibility.

```sql
create table user_conversation_states (
  user_id text not null,
  conversation_id text not null,
  has_read_seq bigint not null default 0,
  last_visible_seq bigint not null default 0,
  muted boolean not null default false,
  archived boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  primary key (user_id, conversation_id)
);
```

Required constraints:

```sql
alter table user_conversation_states
  add constraint user_conversation_states_read_seq_check check (has_read_seq >= 0),
  add constraint user_conversation_states_visible_seq_check check (last_visible_seq >= 0),
  add constraint user_conversation_states_read_visible_check check (has_read_seq <= last_visible_seq);

create index user_conversation_states_user_updated_idx
  on user_conversation_states (user_id, updated_at desc);
```

`unread_count` is derived, not authoritative:

```text
max(0, user_conversation_states.last_visible_seq - user_conversation_states.has_read_seq)
```

For a user with no row, the repository should treat `has_read_seq` as `0` only after membership/visibility is confirmed by message service or conversation membership data. The send path should create or update rows for users that should see the conversation. For single chat this is sender and receiver; for group chat this is the active group-member set supplied by the message service or a later fanout worker.

### message_idempotency_keys

The `messages(sender_id, client_msg_id)` unique constraint is enough to prevent duplicate durable messages. A separate idempotency table is recommended to make conflict checks explicit and to support future pending states if send becomes asynchronous.

```sql
create table message_idempotency_keys (
  sender_id text not null,
  client_msg_id text not null,
  payload_hash text not null,
  server_msg_id text not null,
  conversation_id text not null,
  seq bigint not null,
  status text not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  primary key (sender_id, client_msg_id)
);
```

Required constraints:

```sql
alter table message_idempotency_keys
  add constraint message_idempotency_status_check check (status in ('accepted')),
  add constraint message_idempotency_seq_positive_check check (seq > 0),
  add constraint message_idempotency_server_msg_uniq unique (server_msg_id);
```

Phase 1 can write only `accepted` rows in the same transaction as `messages`. If an insert finds an existing key:

- same `payload_hash`: return the existing `server_msg_id`, `conversation_id`, and `seq` with `deduplicated = true`;
- different `payload_hash`: return idempotency conflict and do not allocate a new seq.

### message_outbox

Stores reliable asynchronous event rows for accepted messages. Detailed schema and repository behavior are defined in [`message-outbox.md`](./message-outbox.md).

The send transaction writes one `message.created` event after the message row, idempotency row, visible conversation states, sender read state, and conversation thread summary are updated. The outbox row is committed atomically with the message and can later be polled by Kafka/Message Transfer/Push workers.

## Sequence Allocation

The repository must allocate seq inside the same PostgreSQL transaction that inserts the message.

Recommended single-node PostgreSQL transaction:

```sql
select max_seq
from conversation_threads
where conversation_id = $1
for update;
```

Then:

```text
next_seq = max_seq + 1
insert messages(...)
update conversation_threads set max_seq = next_seq, last_message_id = ..., last_message_at = ..., updated_at = now()
upsert visible user_conversation_states last_visible_seq to at least next_seq
upsert sender user_conversation_states has_read_seq to at least next_seq
commit
```

For a new conversation:

1. Insert `conversation_threads` with `max_seq = 0`.
2. Lock the inserted row.
3. Allocate `seq = 1`.

The row lock on `conversation_threads` is the serialization point for a conversation. This ensures concurrent sends in the same conversation cannot receive the same seq and cannot skip a seq after commit. Different conversations can allocate seq independently.

Do not allocate seq from Redis in phase 1 durable storage. A future Redis allocator may reserve candidate seq values only if PostgreSQL remains the commit authority and the design accepts possible gaps. The current product contract requires persisted messages to be gap-free per conversation, so PostgreSQL row locking is the default.

## Transaction Boundaries

### CreateMessageIdempotent

One transaction owns idempotency check, seq allocation, message insert, thread update, participant visibility, and sender read-state advance.

Steps:

1. Begin transaction.
2. Derive `conversation_id` and canonical `payload_hash` before database writes.
3. Check `message_idempotency_keys` or `messages` by `(sender_id, client_msg_id)`.
4. If the key exists with matching payload, read and return the existing message; commit or rollback without writes.
5. If the key exists with a different payload, rollback and return idempotency conflict.
6. Upsert and lock `conversation_threads`.
7. Allocate `next_seq = max_seq + 1`.
8. Insert `messages` with `conversation_id + seq` and `sender_id + client_msg_id` unique constraints active.
9. Insert `message_idempotency_keys`.
10. Upsert `user_conversation_states` for every `visible_user_id` with `last_visible_seq = greatest(existing.last_visible_seq, next_seq)`.
11. Upsert `user_conversation_states` for the sender with `has_read_seq = greatest(existing.has_read_seq, next_seq)` and `last_visible_seq = greatest(existing.last_visible_seq, next_seq)`.
12. Update `conversation_threads.max_seq`, `last_message_id`, and `last_message_at`.
13. Insert `message_outbox` with `event_type = message.created`.
14. Commit.

If a unique constraint is hit because of a concurrent retry, the repository should re-read the existing idempotency result and return deduplicated or conflict according to `payload_hash`.

### PullMessages

Pull is a read-only operation:

1. Verify caller participation before or inside repository boundary.
2. Read `messages` by `conversation_id` and inclusive seq range.
3. Order by `seq asc` by default, or `seq desc` when requested.
4. Apply `limit`.
5. Return an empty list for an empty range.

The repository should not advance `has_read_seq` during pull. Read state changes only through send-side sender advancement or explicit mark-as-read.

Pagination validation is shared by memory and PostgreSQL implementations:

- `from_seq`, `to_seq`, and `limit` must not be negative.
- `from_seq = 0` defaults to 1.
- `to_seq = 0` or beyond current max seq is clipped to current max seq.
- `limit = 0` defaults to 50; values above 500 are clipped to 500 and fetched with `limit + 1` to compute `is_end`.
- `order` must normalize to `asc` or `desc` before any SQL is built; invalid or malicious order strings return `INVALID_ARGUMENT`.

### GetConversationSeqStates

This can be a read-only query joining `conversation_threads`, `user_conversation_states`, and optionally the last message:

```text
max_seq = user_conversation_states.last_visible_seq
has_read_seq = coalesce(user_conversation_states.has_read_seq, 0)
unread_count = max(0, last_visible_seq - has_read_seq)
last_message = messages where conversation_id + seq = last_visible_seq
```

When `conversation_ids` is empty, the query should return conversations visible to `user_id` via `user_conversation_states` or future membership tables.

### SetUserHasReadSeqMax

One transaction validates the requested read seq and advances state monotonically:

1. Begin transaction.
2. Lock `user_conversation_states` for `user_id + conversation_id`.
3. Reject when `requested_seq > last_visible_seq`.
4. Upsert `user_conversation_states`:

```sql
has_read_seq = greatest(user_conversation_states.has_read_seq, excluded.has_read_seq)
```

5. Keep `last_visible_seq` at least the resulting `has_read_seq`.
6. Commit and return whether the stored value advanced.

The repository must not decrease `has_read_seq` even if a stale client sends a lower value.

## Redis Model

Redis is optional in phase 1 and must be treated as cache or short-lived coordination only.

Recommended keys:

```text
msg:idempo:{sender_id}:{client_msg_id} -> payload_hash/server_msg_id/conversation_id/seq
msg:thread:{conversation_id}:max_seq -> max_seq
msg:history:{conversation_id} -> sorted set, score seq, value server_msg_id or compact message snapshot
msg:read:{user_id}:{conversation_id} -> has_read_seq
msg:last:{conversation_id} -> last message summary
```

Recommended TTLs:

- idempotency cache: 24 hours or longer than client retry window;
- hot history cache: workload dependent;
- max seq/read state cache: no TTL only if cache invalidation is reliable, otherwise short TTL.

Cache write policy:

- PostgreSQL commit comes first.
- Redis writes happen after commit or through an outbox/worker.
- Cache miss falls back to PostgreSQL.
- Cache conflict never overrides PostgreSQL truth.

## Repository Contract

The storage repository should expose operations equivalent to:

```text
CreateMessageIdempotent(ctx, input) -> message, deduplicated, error
PullMessages(ctx, query) -> messages, is_end, next_seq, error
GetConversationSeqStates(ctx, user_id, conversation_ids) -> states, error
SetUserHasReadSeqMax(ctx, user_id, conversation_id, seq) -> state, updated, error
```

The repository may also expose pure helpers:

```text
SingleConversationID(user_a, user_b) -> conversation_id
GroupConversationID(group_id) -> conversation_id
UnreadCount(max_seq, has_read_seq) -> unread_count
AdvanceReadSeq(current, candidate) -> resulting_has_read_seq
```

Repository errors should distinguish:

- idempotency conflict;
- conversation not found;
- read seq greater than max seq;
- invalid seq range;
- storage unavailable/internal error.

The service layer maps these errors to API/RPC error codes. The repository should not import handler or proto packages.

## Reliability Notes

- The send path is at-least-once at the caller level and exactly-once at storage level for a stable `sender_id + client_msg_id`.
- `conversation_threads` row locking is the consistency point for ordering.
- `messages` and `conversation_threads.max_seq` must commit atomically.
- Sender read-state advancement is part of the send transaction so the sender does not see their own sent message as unread.
- Read-state monotonicity is enforced by both repository logic and database upsert expression.

## Validation

Storage implementation is acceptable when these checks pass:

- duplicate send with same payload returns the original stored message;
- duplicate send with different payload returns idempotency conflict;
- concurrent sends in one conversation produce unique contiguous seq values;
- pull by seq returns messages in requested order without changing read state;
- mark-as-read rejects seq greater than `max_seq`;
- mark-as-read with a lower seq does not reduce `has_read_seq`;
- static verification confirms storage docs are present.

Repository contract tests live in `internal/repository` and run against memory by default:

```bash
go test ./internal/repository
```

PostgreSQL contract tests are opt-in so ordinary local tests do not require middleware. Use a disposable development database with migrations allowed:

```bash
AGENTS_IM_TEST_POSTGRES_CONTRACT=1 \
DATABASE_URL=postgres://agents_im:agents_im_dev_password@localhost:5432/agents_im?sslmode=disable \
go test ./internal/repository
```

`AGENTS_IM_POSTGRES_DSN` may be used instead of `DATABASE_URL`. The opt-in test applies `db/migrations/*.sql` and uses unique message IDs rather than truncating existing data.

## Risks and Tradeoffs

- Row-level locking serializes sends per conversation. This is acceptable for phase 1 and keeps ordering simple, but high-traffic group chats may later require partitioned seq allocation or an append log.
- A separate idempotency table duplicates data already constrained in `messages`, but it makes conflict handling explicit and future asynchronous sends easier.
- Redis can reduce read latency but introduces cache invalidation complexity, so it remains optional and non-authoritative.
