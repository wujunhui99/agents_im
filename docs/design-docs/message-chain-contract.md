# Message Chain Interface Contract and OpenIM-Inspired Design

Status: Draft  
Owner: IM backend  
Related product spec: [`../product-specs/message-chain.md`](../product-specs/message-chain.md)

## Background

The project needs a stable message-chain contract before implementation so independent agents can work in parallel. The contract must cover receiving messages, forwarding/delivery hooks, storage, sync by sequence, and mark-as-read behavior.

OpenIM provides the reference shape:

```text
msggateway -> rpc-msg -> MQ -> msgtransfer -> Redis/cache + Mongo/history + push topic -> push/gateway
```

For agents_im phase 1, we keep the core semantics but implement a simpler synchronous path first:

```text
message-api / message-rpc
  -> validate auth/account/friends/groups
  -> assign conversation seq
  -> store message
  -> update conversation state and read state
  -> write transactional outbox event for future gateway/push
```

## OpenIM design summary

OpenIM's relevant concepts:

- `msggateway` receives WebSocket requests and calls `rpc-msg`.
- `rpc-msg` validates messages and writes message data to MQ using a conversation-based key.
- `msgtransfer` consumes MQ, batches messages by key, assigns seq, writes Redis cache, forwards to Mongo MQ, and forwards to push MQ.
- Mongo consumer persists long-term history.
- Push consumer delivers to online/offline channels.
- Read state is modeled mainly as `userID + conversationID -> hasReadSeq`.
- Mark-as-read operations are monotonic and emit a `HasReadReceipt` notification.

Key implementation ideas to borrow:

1. Conversation-level monotonically increasing seq.
2. Conversation-based MQ key when MQ is introduced.
3. Sender's own read seq advances to the sent message seq.
4. Pull missing messages by `conversation_id + seq` range.
5. Query `has_read_seq` and `max_seq` together for conversation list sync.
6. Treat read receipts as events, not as blocking business logic.

## Phase 1 architecture

### Services involved

#### Message Service

Owns message contract and first implementation.

Responsibilities:

- send message;
- assign `server_msg_id`;
- resolve `conversation_id`;
- assign per-conversation `seq`;
- persist messages;
- maintain conversation `max_seq` and last message summary;
- maintain user read state;
- expose pull/sync/read APIs;
- write `message.created` events into PostgreSQL outbox for future Kafka/Message Transfer/Push integration.

V0 `user_id`, `sender_id`, `receiver_id`, and read-state `user_id` fields are account id aliases. They point to Account Service profiles and are preserved for API/frontend compatibility.

#### Account Service

Used by message service to verify accounts exist. Current code may still use V0 user-rpc/UserLookup names for transport compatibility.

#### Friends Service

Used by message service to validate single-chat policy if the product requires friendship before sending.

Phase 1 contract should include the dependency point but can allow policy to be disabled.

#### Groups Service

Used by message service to validate group existence and membership.

#### Gateway Service, future

Will own WebSocket connection lifecycle and online delivery. It should depend on the message contract, not own message persistence.

#### Message Transfer / Push, future

Will later split asynchronous storage and delivery out of message service.

```text
send_time       int64   // server timestamp in milliseconds
created_at      int64
```

### Ordering and duplicate identity

`conversation_id + seq` is the only authoritative ordering key for confirmed messages. Message Service assigns `seq` while holding the per-conversation storage lock or an equivalent atomic allocation primitive. Downstream Gateway, WebSocket, Kafka, push, and frontend code must treat network arrival order as non-authoritative.

Storage and clients use identity for duplicate suppression:

```text
server_msg_id                // canonical identity after acceptance
sender_id + client_msg_id    // idempotent send identity before/at acceptance
conversation_id + seq        // authoritative position inside a conversation
```

If a repeated delivery has the same `server_msg_id`, it is the same message. If a pending local message has the same `client_msg_id` as an accepted server response from the same sender, the client replaces the pending item with the server snapshot. Reusing the same `sender_id + client_msg_id` with a different payload is an idempotency conflict, not a new message.

`message_origin` is authoritative metadata owned by Message Service. Public REST sends default to `human`; service-to-service Agent writeback can send `ai` with Agent metadata. System status messages use `system`. AI messages remain ordinary stored messages and must still pass idempotency, conversation seq, outbox, pull, and read-state rules.

## RPC contract

Use `internal/rpcgen/message/message.proto` for service-to-service and future gateway integration.

Suggested proto package:

```proto
syntax = "proto3";

package message;

option go_package = "github.com/wujunhui99/agents_im/internal/rpcgen/message/messagepb";

service MessageService {
  rpc SendMessage(SendMessageRequest) returns (SendMessageResponse);
  rpc PullMessages(PullMessagesRequest) returns (PullMessagesResponse);
  rpc GetConversationSeqs(GetConversationSeqsRequest) returns (GetConversationSeqsResponse);
  rpc MarkConversationAsRead(MarkConversationAsReadRequest) returns (MarkConversationAsReadResponse);
}
```

### SendMessage

Request:

```proto
message SendMessageRequest {
  string sender_id = 1;
  string receiver_id = 2;
  string group_id = 3;
  string chat_type = 4;       // single | group
  string client_msg_id = 5;
  string content_type = 6;    // text | image | file
  string content = 7;
  string message_origin = 8;  // service-to-service only; public REST defaults human
  string agent_account_id = 9;
  string trigger_server_msg_id = 10;
  string agent_run_id = 11;
  bool allow_recursive_trigger = 12;
}
```

Response:

```proto
message SendMessageResponse {
  Message message = 1;
  bool deduplicated = 2;
}
```

Behavior:

- `sender_id` must match authenticated user context.
- For single chat, `receiver_id` is required and `group_id` must be empty.
- For group chat, `group_id` is required and `receiver_id` must be empty.
- `client_msg_id` is required.
- `text` content is plain text, non-empty after trim, <= 4096 characters.
- `image` content is JSON with at least `mediaId`; the media record must be owned by the sender, `purpose=message_image`, `status=ready`, `content_type=image/*` from the allowlist, and <= 15 MiB.
- `file` content is JSON with `mediaId`, `filename`, `sizeBytes`, and `contentType`; the media record must be owned by the sender, `purpose=message_file`, `status=ready`, <= 20 MiB, and the message metadata must match the media record.
- Same `sender_id + client_msg_id` returns the prior message if payload matches.
- Different payload for same idempotency key returns idempotency conflict.

### PullMessages

Request:

```proto
message PullMessagesRequest {
  string user_id = 1;
  string conversation_id = 2;
  int64 from_seq = 3;
  int64 to_seq = 4;
  int32 limit = 5;
  string order = 6; // asc | desc, default asc
}
```

Response:

```proto
message PullMessagesResponse {
  repeated Message messages = 1;
  bool is_end = 2;
  int64 next_seq = 3;
}
```

Behavior:

- Caller must be a participant.
- `from_seq` and `to_seq` are inclusive.
- `limit` protects large pulls.
- Empty result is valid.

### GetConversationSeqs

Request:

```proto
message GetConversationSeqsRequest {
  string user_id = 1;
  repeated string conversation_ids = 2;
}
```

Response:

```proto
message ConversationSeqState {
  string conversation_id = 1;
  int64 max_seq = 2;
  int64 has_read_seq = 3;
  int64 unread_count = 4;
  int64 max_seq_time = 5;
  Message last_message = 6;
}

message GetConversationSeqsResponse {
  repeated ConversationSeqState states = 1;
}
```

Behavior:

- If `conversation_ids` is empty, return all visible conversations for the user.
- `unread_count = max(0, max_seq - has_read_seq)`.

### MarkConversationAsRead

Request:

```proto
message MarkConversationAsReadRequest {
  string user_id = 1;
  string conversation_id = 2;
  int64 has_read_seq = 3;
}
```

Response:

```proto
message MarkConversationAsReadResponse {
  string conversation_id = 1;
  int64 has_read_seq = 2;
  int64 max_seq = 3;
  int64 unread_count = 4;
  bool updated = 5;
}
```

Behavior:

- Caller must be a participant.
- Reject `has_read_seq > max_seq`.
- Update is monotonic.
- Emits a future-compatible `message.read` event when updated.

## Event contract for future parallel work

Even if phase 1 does not implement Kafka/gateway, the contract should define event shapes.

Kafka-compatible topic, schema, producer abstraction, and delivery semantics are versioned in [`kafka-message-events.md`](./kafka-message-events.md). The examples below remain the phase-1 conceptual shape; the canonical transport schema is `MessageEvent`.

### message.accepted

Emitted after message is persisted.

```json
{
  "eventId": "evt_...",
  "eventType": "message.accepted",
  "conversationId": "single:user_a:user_b",
  "seq": 1,
  "serverMsgId": "...",
  "senderId": "user_a",
  "receiverIds": ["user_b"],
  "createdAt": 1710000000000
}
```

### message.read

Emitted after `has_read_seq` advances.

```json
{
  "eventId": "evt_...",
  "eventType": "message.read",
  "conversationId": "single:user_a:user_b",
  "userId": "user_b",
  "hasReadSeq": 10,
  "readAt": 1710000000000
}
```

### Delivery ownership

- Message service emits accepted/read events.
- Gateway/push service owns online delivery and ACK.
- Storage worker, when introduced, owns asynchronous durable storage.
- Message service must not depend on gateway internals.

## Media dependency

Media metadata is stored in PostgreSQL `media_objects`; object bytes live in RustFS (S3-compatible) storage. Message send does not trust client-provided object keys or URLs. For image/file messages it validates only a `mediaId` reference and requires the media record to be ready and owned by the sender before persisting the message.

Download URL authorization for `/media/:media_id/download-url` is owner-only for non-message media. Image/file message attachments also allow conversation participants who can see the message containing the referenced `mediaId`; non-uploader clients must pass that message's `serverMsgId` as `msg_id` so media-api can validate that the message actually references the requested media.

## Storage contract

Phase 1 can use an in-memory repository, but repository interfaces should match persistent storage behavior.

Required operations:

```text
CreateMessageIdempotent(sender_id, client_msg_id, payload) -> Message, deduplicated
NextConversationSeq(conversation_id) -> seq
SaveMessage(message)
GetMessages(conversation_id, from_seq, to_seq, limit, order) -> []Message
GetConversationMaxSeq(conversation_id) -> max_seq
GetUserHasReadSeq(user_id, conversation_id) -> has_read_seq
SetUserHasReadSeqMax(user_id, conversation_id, seq) -> updated bool
GetConversationSeqStates(user_id, conversation_ids) -> []ConversationSeqState
```

Persistent implementation should enforce:

```text
unique(conversation_id, seq)
unique(sender_id, client_msg_id)
unique(server_msg_id)
```

The PostgreSQL implementation allocates seq by upserting/selecting the `conversation_threads` row `for update`, then inserting the message and idempotency row in the same transaction. A unique-constraint race on `sender_id + client_msg_id` must resolve to the existing message only when the payload hash matches; otherwise it must surface an idempotency conflict. The failed racing transaction must not commit a consumed seq.

Pull APIs must return messages ordered by numeric `seq` for the requested direction. `last_message` and `max_seq_time` in conversation state must reflect the visible max seq row, not the newest wall-clock `send_time`.

Client sync responsibilities:

- Sort confirmed messages by `conversation_id + seq`.
- Deduplicate by `server_msg_id`, then by `client_msg_id` for local pending replacement.
- Keep pending local messages without seq after confirmed messages in local enqueue order.
- On detected seq gaps, pull the missing range from the highest local contiguous seq plus one through the server `max_seq`.
- Serialize or queue same-conversation local sends, or disable the composer until the previous send is accepted or failed.

## Validation and dependencies

### Single chat send

Message service needs:

- `user` check sender and receiver exist;
- optional `friends` check relation allows sending.

### Group chat send

Message service needs:

- `groups` check group exists;
- `groups` check sender is group member;
- `groups` list group member IDs for future delivery fanout.
- `media` check for image/file message owner, purpose, ready status, content type, and size.

### Pull/read

Message service must verify caller participates in conversation.

For phase 1 this can be repository-based once conversations are created by send path.

## Parallel development boundaries

This contract enables these branches to be developed in parallel:

1. `feature/message-service-contract`
   - `internal/rpcgen/message/message.proto`
   - `api/message.api`
   - message logic skeleton
   - memory repository and tests

2. `feature/message-storage`
   - repository interfaces
   - PostgreSQL schema draft or memory implementation
   - idempotency and seq allocation tests

3. `feature/gateway-contract`
   - WebSocket command mapping to `SendMessage`, `PullMessages`, `GetConversationSeqs`, `MarkConversationAsRead`
   - no message persistence logic

4. `feature/read-receipts`
   - `message.read` event plumbing
   - monotonic read-state tests

Recommended order:

```text
message-service-contract first
then message-storage, gateway-contract, read-receipts can proceed in parallel
```

## Testing requirements

Minimum tests for the first implementation:

- single chat send creates conversation and seq 1;
- repeated `sender_id + client_msg_id` returns deduplicated response;
- different payload with same idempotency key fails;
- group chat send requires membership;
- pull messages returns ascending seq range;
- sender `has_read_seq` advances after send;
- mark as read rejects seq greater than max seq;
- mark as read is monotonic;
- conversation seq query returns `max_seq`, `has_read_seq`, `unread_count`.

## Open questions

- Whether single chat requires existing friendship in phase 1 or only user existence.
- Whether message service owns conversation list fully or a separate conversation service is introduced after phase 1.
- Whether read receipts should be visible to clients before WebSocket gateway is implemented.
- Whether `content` should remain string in phase 1 or become JSON bytes in proto.
