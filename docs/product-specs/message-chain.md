# Message Chain Product Spec

Status: Draft  
Owner: IM backend  
Reference: OpenIM message path, simplified for agents_im phase 1

## Goal

Build the first message-chain contract for agents_im so message, gateway, storage, and read-state work can be developed in parallel.

This spec covers user-visible behavior for:

- sending messages;
- storing messages;
- pulling messages by sequence;
- querying conversation read/max sequence state;
- marking messages or conversations as read.

## Non-goals for phase 1

The first contract intentionally does not require:

- WebSocket gateway implementation;
- Kafka or message-transfer worker implementation;
- offline push notification implementation;
- per-device sync state;
- message recall/delete;
- reactions, edits, typing, mentions, rich media processing;
- end-to-end encryption;
- distributed database sharding.

These should remain compatible with the interface contract but are not part of the first implementation.

## Users and actors

- Sender: authenticated user sending a message.
- Receiver: user receiving a single-chat message.
- Group member: user receiving or sending messages in a group.
- Client: mobile/web/desktop app. Phase 1 can use HTTP APIs instead of WebSocket.
- Message service: service owning message sequence, message persistence, and read state.

## Chat types

Phase 1 supports:

1. Single chat: user-to-user messages.
2. Group chat: user-to-group messages.

## Message content

Phase 1 supports text messages only at product level.

Future message content types should be versioned through `content_type` and `content` without changing sequence/read contracts.

Text `content` must be non-empty after trimming and must be 4096 characters or fewer.

## Message identity

Each message has two IDs:

- `client_msg_id`: generated by client or caller for idempotency.
- `server_msg_id`: generated by message service and globally unique.

`client_msg_id` must be non-empty, 128 characters or fewer, and must not contain NUL.

Product behavior:

- Retrying a send request with the same `sender_id + client_msg_id` must not create duplicate messages.
- The server should return the existing message result when the duplicate request is semantically identical.
- If the same `client_msg_id` is reused with different payload, the server should reject it as an idempotency conflict.

## Conversation identity

Each message belongs to one conversation.

Phase 1 conversation ID rules:

- Single chat: deterministic from the two user IDs, independent of sender order.
- Group chat: deterministic from `group_id`.

Example logical forms:

```text
single:{min_user_id}:{max_user_id}
group:{group_id}
```

The exact string format is a design detail, but it must be stable and documented.

The `sender_id`, `receiver_id`, and `group_id` components used to derive a conversation ID must be canonical service IDs, not user-facing identifiers. Phase 1 accepts non-empty components up to 128 characters, forbids `:` and NUL because `:` is the conversation ID delimiter, and rejects any derived `conversation_id` longer than 256 characters. The system-generated `usr_...` and `grp_...` IDs satisfy these constraints.

## Sequence model

Each conversation has a monotonically increasing `seq`.

Product behavior:

- `seq` starts at 1 for the first persisted message in a conversation.
- `seq` increases by 1 for each persisted message in that conversation.
- `seq` is unique only within a conversation.
- Clients use `conversation_id + seq` to pull and de-duplicate messages.

## Send message behavior

When a user sends a message, the system must:

1. Authenticate the sender with JWT Bearer token and use token `user_id` as `sender_id`.
2. Validate the target:
   - single chat target user exists;
   - group chat group exists;
   - group sender is a member of the group.
3. Optionally validate social rules:
   - single chat can later require friendship depending on product policy.
4. Resolve `conversation_id`.
5. Enforce idempotency using token-derived `sender_id + client_msg_id`; if the request body includes `senderId`, it must match the token user.
6. Assign `server_msg_id` and conversation `seq`.
7. Persist the message.
8. Update conversation `max_seq` and last message summary.
9. Set sender read state to at least the sent message `seq`.
10. Return the accepted message metadata.

Phase 1 can complete all steps synchronously in the message service.

## Send response behavior

A successful send response should include:

- `server_msg_id`;
- `client_msg_id`;
- `conversation_id`;
- `seq`;
- `send_time`;
- persisted message snapshot.

The response means the message has been accepted and stored by the message service.

It does not guarantee that every recipient has received it online.

## Pull messages behavior

Clients can pull messages by conversation and seq range.

Behavior:

- Caller must be a participant of the conversation.
- The service returns messages in ascending seq order by default.
- `order` must be `asc`, `desc`, or omitted.
- `limit` defaults to 50 when omitted or zero and is clipped to 500.
- Missing/deleted future messages are out of scope for phase 1.
- Pulling an empty range returns an empty list, not an error.

Common client sync flow:

```text
client local seq < server max seq
=> PullMessages(conversation_id, from_seq = local_seq + 1, to_seq = server max_seq)
```

## Conversation seq query behavior

Clients can query conversation state:

- `max_seq`: latest message seq in that conversation.
- `has_read_seq`: latest seq this user has marked as read.
- `unread_count`: derived as `max_seq - has_read_seq`, never below zero.
- `last_message`: optional summary for conversation list.

If a user has never read a conversation, `has_read_seq` defaults to 0.

## Mark as read behavior

The system supports marking an entire conversation as read up to a seq.

Behavior:

1. Caller must be a participant of the conversation.
2. `has_read_seq` must be greater than or equal to 0.
3. `has_read_seq` must not exceed current `max_seq`.
4. The update is monotonic:
   - if new seq is less than current `has_read_seq`, keep current value;
   - if new seq is greater, update it.
5. Return the resulting `has_read_seq`, `max_seq`, and `unread_count`.

## Per-message read receipts

Phase 1 prioritizes conversation-level read state.

Single-chat read receipts can be represented as events later:

```text
MessageReadEvent(user_id, conversation_id, has_read_seq, read_at)
```

The first implementation may store or expose this event only through tests or an internal repository. WebSocket push of read receipts is a later gateway task.

## Error cases

The interface should distinguish:

- unauthenticated or invalid token;
- sender mismatch;
- user not found;
- group not found;
- not a group member;
- conversation not found or caller not a participant;
- invalid content;
- invalid seq range;
- read seq greater than max seq;
- idempotency conflict.

## Acceptance criteria

The first message contract is ready when:

- product spec and design docs define the API/RPC contract;
- request/response schemas are stable enough for parallel agents;
- message, gateway, and storage agents can implement against the same contract;
- read state uses `user_id + conversation_id -> has_read_seq`;
- send and read behavior are idempotent/monotonic;
- the contract clearly separates phase 1 synchronous implementation from future MQ/gateway work.
